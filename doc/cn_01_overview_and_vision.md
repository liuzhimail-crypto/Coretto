# 总览与愿景

## 1. 这个项目是什么

这个项目最初来自一个很现实的目标：**减少昂贵的云端 token 消耗**。思路是把 agent 任务拆开，把最耗 token、但结构上可管理的部分留给本地 14B 模型，把真正必要的部分再交给云端模型。

但设计方向后来已经超出了狭义的路由工具。

现在，这个系统更适合被定义为：

> **一个可进化的广域信息服务体系**

而不是一个单纯的 token router。

## 2. 它真正的 ambition

这个系统想实现的是：

1. **理解** 输入进来的信息；
2. **提取并压缩** 其中真正重要的部分；
3. **保留高价值残差**——即那些当前本地模板无法安全吸收的信息；
4. **只在必要时** 才把更强的解释任务交给云端；
5. **从打破模板的案例中学习**，让本地 schema 随时间演化。

所以它不是在追求“更好的摘要”，而是在追求：

> 把原始、混沌、异质的输入，编译成可服务、可推理、可交接的信息核心。

## 3. 本地模型的角色

本地 14B 不是一个廉价替代模型，而是：

- 请求编译器，
- 背景压缩器，
- 宪法过滤器，
- 异常保留器，
- 本地智能前端。

它的职责，是为后续推理生成 **最小充分背景**。

## 4. 云端模型的角色

云端模型不是默认的第一处理器。它主要在以下场景被调用：

- 更强的推理，
- 更强的解释，
- 合宪审查，
- 模板外高价值信息的语义补全，
- 判断本地提炼出的颗粒信息是否真的构成可信知识体系。

## 5. 这个系统承认的核心张力

这个项目明确接受一个基础事实：

- 本地 schema 是有限的；
- 真实世界语义比任何当前模板都更宽；
- 因此封闭 schema 永远不够。

所以系统必须同时具备：

- **结构化压缩**，以及
- **开放残差保留**。

这正是它“可进化”而不是“脆弱僵硬”的原因。

## 6. 它和普通系统的区别

它不只是：

- provider routing，
- model switching，
- prompt truncation，
- 或普通摘要。

它更接近一个：

> **自路由 + 自压缩 + 残差保留的信息编译器**

由本地模型负责判断：

- 什么信息是真正高价值，
- 什么能在本地被安全服务，
- 什么必须保持开放，
- 什么应该上交云端，
- 以及何时它已经不再信任自己的全局综合能力。

## 7. 当前一句话身份

> 一个 local-first 的信息编译器：提取影响决策的核心，保留高价值异常，只把真正需要云端解释的部分升级上去。
