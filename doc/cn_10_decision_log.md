# 决策日志

本文件压缩记录对话中已经确认的重要决策。

## 已确认决策

### 1. 项目不只是 token routing
这个系统被定义为 **可进化的广域信息服务体系**，而不只是路由器。

### 2. 本地 14B 是主智能前端
它负责：

- 任务拆解，
- 本地信息提取，
- 压缩，
- 结构化表示，
- 残差保留，
- 能力边界感知。

### 3. LLM-first 是硬原则
只要 LLM 能通过理解、压缩、结构化或自评解决问题，就不要过早引入额外模块。

### 4. 开放残差保留是强制要求
所有无法安全进入本地 schema 的高价值信息，都必须保留显式出口。

### 5. 压缩目标是最小充分背景
本地模型不是写漂亮总结，而是保留那些会影响下游推理的内容。

### 6. KIR-Lite 是第一版运行中的中间表示
KIR-Lite v1.1 是当前工作 schema。

### 7. 能力边界必须显式表示
本地模型必须识别超载，并从伪全局综合切换到颗粒保全模式。

### 8. 云端不仅求解，还要审查
云端模型可以检查本地结构化输出是否真的形成可信知识体系。

## 已澄清的关键概念

### 高价值信息
任何会改变答案、推理路径、约束满足、风险判断或演化方向的信息。

### 可服务信息
任何当前本地 schema 能安全表达、压缩和传递的信息。

### 可进化残差
高价值、但当前本地 schema 无法安全吸收、同时又不能丢失的信息。

## 暂未完全解决的问题

这些内容在对话中尚未完全定义，属于后续工作：

1. 训练或评估时应如何精确定义 capacity overload？
2. KIR-Lite 应保持纯 prompt 驱动，还是后续转为 SFT target？
3. 云端 solver prompt 与云端 audit prompt 应如何分化？
4. 应采用怎样的数据格式来训练本地 14B 稳定输出 KIR-Lite？
5. 除 token 减少之外，成功压缩还应使用哪些指标？
6. 重复出现的 residual pattern 应如何升级为未来 schema？

## 当前最推荐的下一步

拿几段真实背景材料直接跑本地 prompt，重点观察模型在以下字段上的漂移：

- `capacity_guard`
- `open_exit`
- `must_keep`
- `dynamic_constraints`
- `cloud_audit` 触发条件

这会直接暴露：这个表示层到底已经可用，还是仍然过于激进。
