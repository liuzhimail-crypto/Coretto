# 架构

## 1. 系统分工

这个架构是明确的 local-first。

### 本地 14B 模型
扮演：

- 信息编译器，
- 任务核心提取器，
- 背景压缩器，
- 本地宪法裁判，
- 残差保留器，
- 能力边界自检器。

### 云端模型
扮演：

- 更强的推理器，
- 更强的解释器，
- 合宪审查器，
- 模板外高价值信息的解释器，
- 判断本地颗粒是否构成可信知识体系的检查者。

## 2. 主处理链

### 正常路径
原始背景
→ 本地 14B 编译成 KIR-Lite v1.1
→ 如果 `cloud_audit.required = false`
→ 把 `cloud_handoff` 交给云端 solver
→ 云端完成最终推理

### 复杂 / 超载路径
原始背景
→ 本地 14B 编译成 KIR-Lite v1.1
→ `capacity_guard.status = borderline | overflow` 或 `cloud_audit.required = true`
→ 云端先做合宪审查
→ 然后选择以下之一：
- 直接求解，
- 要求重编译，
- 重新裁切后再提交，
- 把部分残差提升为主 schema 重点关注对象。

## 3. 为什么这种架构重要

它避免了两种典型失败：

### 失败 A —— 把所有原文直接送云端
这会浪费 token 预算，并让云端模型承担低层清洗工作。

### 失败 B —— 强迫本地模型假装自己已经完整理解
这会产生“看起来干净整齐、实际上不可信”的本地知识结构。

这个设计要求系统显式执行：

- 本地压缩，
- 本地 schema 感知提取，
- 显式残差保留，
- 显式超载意识，
- 在必要时进行云端审查。

## 4. 颗粒保全模式

当检测到超载时，本地模型应停止构造漂亮的全局理论，转而进入 **颗粒保全模式**。

此时它优先处理：

- task core，
- key parameters，
- hard constraints，
- 局部可信规则，
- must-keep 片段，
- 开放残差，
- 裁切建议。

这会在压力下保住可信性。

## 5. 审查是一种二阶推理

云端审查不只是 validation。
它是一种二阶推理，核心问题包括：

- 本地表示是否足够完整？
- 有没有隐藏冲突？
- 是否存在明显误分类？
- 本地模型是否在超载状态下伪装成全局理解？
- 有没有值得未来吸收的 schema-breaking 模式？

## 6. 演化闭环

这就是系统“可进化”的来源。

原始输入
→ 本地提取
→ 云端审查 / 解释
→ 识别重复出现的残差模式
→ 成为未来 schema 演化候选
→ 增强未来的本地可服务能力

也就是说，系统是通过学习残差信道来进化的。
