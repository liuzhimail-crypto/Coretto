# Engineering Evolution Summary (2026-03-21 ~ 2026-03-22)

## 总览
这两天的工程演化，核心不是“再发明一个更复杂系统”，而是把一个逐渐成形的 prompt-driven local compiler 体系，压成一个可测试、可回归、可验收的工程对象。

演化主线可以概括为五步：

1. **宪法与中间表示确立**
2. **prompt 反复收敛**
3. **回归测试体系建立**
4. **输出清洗与评测链路打通**
5. **工程基线正式冻结**

---

## Day 1 — 2026-03-21：从愿景到结构

### 1. 系统愿景开始收敛为“本地智能编译器”
项目最初的目标，不只是做 token 节省，也不是传统 route evaluator，而是让本地 14B 直接承担：

- 请求结构理解
- 背景信息压缩
- 高价值信息识别
- 模板外高价值残差保留
- 必要时把强推理与审查交给云端

这一阶段最重要的收敛不是工程实现，而是**世界观**：
本地模型不是简单 router，而是 **query-to-knowledge compiler**。

### 2. 两条设计原则被明确
这两条原则最终贯穿了后续所有 prompt 版本：

- **LLM 优先**  
  能由模型自身理解、压缩、结构化、自评完成的，不优先发明额外模块。

- **开放出口保护**  
  当高价值信息无法安全落入模板时，不强塞；保留到 `open_exit`，交给云端解释。

### 3. KIR-Lite 作为中间表示成形
为适配本地 14B 的实际生成能力，复杂体系被压成了 KIR-Lite：

- `task_core`
- `key_params`
- `static_constraints`
- `dynamic_constraints`
- `mechanisms`
- `must_keep`
- `open_exit`
- `capacity_guard`
- `cloud_handoff`
- `cloud_audit`

这一步的意义是：  
**从抽象愿景进入可执行结构。**

### 4. 三个核心定义被明确
“宪法”开始具体化为三种信息判定：

- **高价值信息**
- **可服务信息**
- **可进化残差**

随后又进一步被压成 14B 可执行判令，并被嵌入 system prompt。

## Day 2 — 2026-03-22：从结构到工程基线

### 1. 从 v1.1 到 v1.3c 的持续收敛
system prompt 经历了连续版本收敛，大致逻辑如下：

#### v1.1
建立完整哲学框架，但存在：
- 裸 JSON 不稳
- `task_core` 元任务化
- `cloud_handoff` 自由发挥
- `capacity_guard` 缺乏可观察证据
- 复杂样本上 `overflow` 不够果断

#### v1.2 / v1.2a
主要修：
- 输出纪律
- `task_core` 去元任务化
- `open_exit` / `must_keep` / `static_constraints` 去重
- `capacity_guard.reasons` 具体化
- `cloud_handoff` 不再发明新方案

#### v1.3
进入工程稳定性修补：
- 强化 schema 纪律
- 强化 handoff 忠实性
- 提高复杂样本上的保守审查
- 防止 `overflow` 时整体塌缩

#### v1.3a / v1.3b
不再做大改，只补：
- 风险校准
- `open_exit` 与状态判断联动
- handoff 再缩短
- `cloud_focus` 再收紧

#### v1.3c
收口到最后两个顽固问题：
- 边界意识任务默认更保守
- `cloud_focus` 硬上限锁定

### 2. 测试集从泛测走向定向回归
测试集不再只是“大概试试”，而是逐步按缺陷点拆分：

- JSON 纪律测试
- open_exit 漏检测试
- handoff 忠实性测试
- overflow 颗粒保全测试
- 槽位纯度与去重测试
- compound / boundary / risk calibration 窄测试

这一步让工程开始从“主观评估”进入“问题定向打靶”。

### 3. 最困难的问题逐渐从 prompt 转移到工程链路
后期暴露出的最麻烦问题不再只是 prompt 行为，而是：

- Ollama 输出的 fenced JSON
- 脏输出恢复失败
- schema 漂移
- gate / observe / stress 目录混淆
- 单次随机漂移导致误判

这推动工程重心转向：
**先修评估回路，再继续调 prompt。**

### 4. baseline engineering pack 出现
随着 `v1.3c` 基本收敛，工程策略改变为：

- 不再继续无基线地试 prompt
- 先冻结当前可工作版本为 baseline
- 给 baseline 配套：
  - gate
  - observe
  - stress
  - 清洗工具
  - schema 验证
  - 验收标准
  - 多数表决

### 5. v2 工程包修正了评测层 bug
最终通过 `coretto_v1_3c_baseline_engineering_pack_v2` 解决了三个关键工程问题：

- Gate 输出隔离
- 分套件独立报告
- Gate 三次多数表决

这一步的意义非常大：  
**它把“单次结果”变成了“工程基线”。**

## 截止 2026-03-22 的最终工程结论

### 已经完成的
- 系统宪法已明确
- KIR-Lite 已稳定
- 本地 compiler 哲学已固化
- v1.3c prompt 已可工作
- baseline engineering pack 已建立
- gate 多数表决通过

### 尚未完成的
- 更大规模真实任务分布上的长期稳定性验证
- 更广 domain migration 的系统对比
- benchmark 指标的进一步 formalization
- baseline 与未来版本的自动差异分析

## 建议的下一阶段工作模式
从这个里程碑之后，建议采用以下工程模式：

1. **冻结当前基线**
2. **所有后续版本新起版本号**
3. **先过 gate，再看 full eval，再看 hammer**
4. **先做工程比较，再做哲学升级**
5. **减少 prompt 大改，增加可解释对比**
