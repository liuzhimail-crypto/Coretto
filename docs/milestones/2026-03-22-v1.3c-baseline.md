# Milestone: v1.3c Baseline Established (2026-03-22)

## 状态
已建立可工作的工程基线：

- **Prompt 基线**：`v1.3c`
- **工程基线包**：`coretto_v1_3c_baseline_engineering_pack_v2`
- **Gate 策略**：3 个 gate 样本 + 3 次多数表决
- **当前结论**：Gate 通过，可作为后续工程迭代与回归比较基线

## 里程碑意义
这是 Coretto 从“连续 prompt 试验”转向“有基线的工程迭代”的分水岭。

在这个节点之前，主要工作集中在：
- 明确系统宪法
- 固化 KIR-Lite v1.1
- 反复修正 system prompt
- 构建本地 14B 的知识压缩 / 风险校准 / handoff 纪律
- 修复 Ollama 输出清洗、schema 漂移与回归评估链路

在这个节点之后，工程重点应转向：
- 基线冻结
- 批量 hammer 测试
- 更真实任务分布下的稳定性观察
- 在 baseline 之上做小步版本比较，而不是继续无基线试错

## 当前基线的核心行为
当前基线能够较稳定地做到：

1. **边界意识任务默认更保守**
   - 对边界意识 / 审查判断 / 高风险研究判断任务，不再轻易输出 `safe`
   - 更倾向于 `borderline + audit=true`

2. **handoff 长度与忠实性被压住**
   - `cloud_focus` 受硬上限约束
   - handoff 不再像早期版本那样自由发挥成新的设计问题

3. **open_exit 能影响风险判断**
   - 高价值难归类信息不再只是被看见，而会真实影响状态判断与审查偏置

4. **工程评测链路基本打通**
   - 可自动抽取 / 清洗 Ollama 输出
   - 可分层运行 gate / observe / stress
   - 可做多数表决，减少单次漂移的误导

## 当前仍保留的工程现实
这不是“完美版本”，而是“可工作的基线”。

仍需持续观察的点包括：
- 更长、更脏、更多真实任务输入下的稳定性
- stress 层是否出现风险校准回摆
- schema 漂移是否在长跑中重新出现
- `safe / borderline / overflow / audit=true` 的分布是否足够稳定

## 建议的后续工程纪律
1. 不要覆盖当前基线，后续版本一律新起版本号
2. 任何 prompt 改动，先跑 gate，再跑 full eval，再和 baseline 对比
3. 先保 baseline 稳定，再考虑进入 `v1.3d` / `v1.4`
4. 保持 prompt 改动小步化，优先工程对比，而不是重新大改哲学
